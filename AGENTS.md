# AGENTS.md

## Обзор проекта

Фронтенд проекта построен на Vue 3 с использованием сборщика Vite. Для стилей применяется SCSS; управление состоянием реализовано через Pinia; маршрутизация — с помощью Vue Router; HTTP-запросы выполняются библиотекой axios; для ленивой загрузки изображений подключен vue-lazyload; для создания слайдеров используется Swiper. Бэкенд-часть в репозитории пока отсутствует; при необходимости серверную логику следует реализовать на Node.js (например, с использованием Express) и обеспечить взаимодействие фронтенда с API через axios.

В проект интегрирован AI-агент (OpenAI Codex) — виртуальный full-stack разработчик с выдающимся опытом. Агент всегда ищет креативные пути решения задач и стремится оптимизировать код для максимальной скорости и надежности, избегая багов. Все общение с пользователем агент ведет на русском языке.

## Установка и запуск

- Установи зависимости проекта командой `npm install`.
- Запусти режим разработки командой `npm run dev` (локально приложение станет доступно по адресу `http://localhost:5173/`).
- Выполни сборку проекта для продакшна командой `npm run build` (результат появится в директории `dist/`).
- Для локальной проверки production-сборки используй `npm run preview` (если эта команда доступна).
- При добавлении серверной части убедись, что для её запуска определены соответствующие команды (например, `npm run server`), и опиши их в документации.

## Тестирование

- Убедись, что перед коммитом все тесты проходят успешно. При падении тестов исправь код и повторно запусти тесты до полного прохождения.
- Если система непрерывной интеграции (CI) настроена, ознакомься с конфигурацией в файлах `.github/workflows/` и убедись, что твои изменения удовлетворяют всем указанным проверкам.
- Добавляй или обновляй тесты для любого нового функционала или исправляемых багов, даже если это не было явно запрошено.
- Если автоматических тестов нет, тщательно протестируй изменения вручную: запусти приложение локально (`npm run dev`) и проверь основные сценарии работы нового или измененного функционала.

## Стиль кода

- **Комментарии:** Не пиши очевидные комментарии в коде только там где это действительно необходимо.
- **Именование:** Используй понятные английские названия для переменных, функций, компонентов и файлов, отражающие их назначение.
- **Форматирование:** Придерживайся установленного в проекте стиля форматирования кода (см. конфигурации ESLint/Prettier, если есть). Обычно используются единый отступ (например, 2 пробела), единый стиль кавычек и прочие правила код-стайла. Автоматические инструменты форматирования (Prettier) запускай перед коммитом, чтобы исправить стиль.
- **Структура кода:** Разделяй логику на небольшие переиспользуемые функции и компоненты. Избегай дублирования кода.
- **Стили (SCSS):** Организуй файлы стилей логично. Используй возможности SCSS (например, переменные, вложенность) для улучшения читаемости. Придерживайся единого подхода к именам CSS-классов (например, методология BEM для структурирования имен).
- **Использование библиотек:** Применяй подключенные библиотеки по назначению, не изобретай велосипед:
    - *Pinia:* Храни глобальное состояние только в Pinia-сторах. Не используй глобальные переменные или `$emit` для обмена данными между компонентами.
    - *Vue Router:* Управляй навигацией через роутер (например, используй `router.push()` для переходов). Определяй маршруты в файлах конфигурации роутера, избегай жестко прописанных URL в коде компонентов.
    - *Axios:* Выполняй HTTP-запросы через `axios`. Централизуй настройки (базовый URL, таймауты) при необходимости. Обрабатывай ошибки запросов (try/catch) и информируй пользователя о проблемах (например, показывай уведомления об ошибках).
    - *vue-lazyload:* Для изображений используй механизм ленивой загрузки, предоставляемый этой библиотекой, вместо ручного отслеживания загрузки изображений.
    - *Swiper:* Для реализации слайдеров и каруселей используй библиотеку Swiper, а не создавай такие компоненты с нуля.
- **Зависимости:** Не добавляй новые внешние зависимости без острой необходимости. Старайся решить задачу средствами текущего стека технологий.
- **Безопасность:** Следуй лучшим практикам безопасного кодирования: проверяй и фильтруй входные данные пользователей, избегай вставки неэкранированных данных в DOM (для предотвращения XSS), не храни чувствительные данные (пароли, токены) в репозитории открыто.
- **Оптимизация:** Пиши максимально эффективный код. Избегай неоправданно сложных операций в циклах или шаблонах Vue. Используй ленивую загрузку модулей или компонентов, когда это улучшает производительность. Стремись к тому, чтобы приложение оставалось быстрым и отзывчивым даже при увеличении объема данных.
- **Чистота кода:** Перед завершением работы убери отладочный код: удали вызовы `console.log` и неиспользуемые фрагменты кода.

## Выполнение задач

При решении задач соблюдай следующий алгоритм, действуя вдумчиво и последовательно:

1. **Анализ задачи.** Внимательно прочитай формулировку задачи/проблемы и изучи соответствующий код проекта. Удостоверься, что полностью понимаешь, какие изменения или новая функциональность требуются.
2. **Уточнение требований.** Если какие-либо детали или требования неясны, запроси уточнения у пользователя (на русском языке) перед тем, как приступать к реализации.
3. **Декомпозиция.** Если задача крупная или сложная, разбей её на более мелкие подзадачи. Выполняй их поочередно, проверяя промежуточные результаты на каждом этапе.
4. **Планирование решения.** Продумай подход к реализации: какие модули или файлы потребуются для изменений, какие компоненты или функции нужно создать или изменить. Наметь структуру решения прежде, чем писать код.
5. **Реализация.** Напиши необходимый код для решения задачи. Вноси изменения на фронтенде (Vue-компоненты, состояние Pinia, маршруты, стили) и/или на бэкенде (если требуется создать API или серверную логику) — исходя из поставленной задачи. Соблюдай при этом требования раздела «Стиль кода» и эффективно используй имеющиеся библиотеки.
6. **Тестирование изменений.** После реализации запусти тесты (`npm run test`) и убедись, что все они проходят успешно. Дополнительно протестируй новый или измененный функционал вручную в браузере, чтобы проверить корректность работы.
7. **Отладка.** Если обнаружены ошибки или тесты не проходят, проанализируй проблему, внеси исправления в код и повторно запусти тесты. Продолжай итерации правок и тестирования, пока проблема не будет устранена.
8. **Оптимизация.** Убедившись в правильности решения, оцени код с точки зрения эффективности. По возможности проведи рефакторинг для повышения производительности и читабельности кода, не нарушая его работоспособности.
9. **Документация.** Если твои изменения влияют на процесс установки, запуска или использования приложения, обнови соответствующую документацию (например, README.md).
10. **Коммит.** Подготовь коммит с внесёнными изменениями. Просмотри `git diff`, чтобы убедиться, что включены только необходимые правки. Составь понятное сообщение коммита на русском языке, описывающее суть изменений (см. раздел ниже о коммитах).
11. **Язык задач** Задачи пиши на русском языке.

## Коммиты и Pull Request

- Пиши сообщения коммитов на английском, чётко и информативно резюмируя сделанные изменения. Начинай сообщение с глагола в повелительном наклонении, описывающего действие (например: «Добавить поддержку загрузки изображений…», «Исправить ошибку в компоненте X…»).
- Если коммит исправляет баг или добавляет новую функцию, укажи это в сообщении (например: «Исправить некорректное отображение списка…»). При необходимости упомяни ID соответствующего issue или задачи.
- Старайся, чтобы один коммит содержал логически цельные изменения, связанные с одной задачей. Не смешивай в одном коммите правки по несвязанным функциональностям.
- Перед созданием коммита убедись, что проект успешно собирается, все тесты проходят и линтер не выдаёт ошибок.
- Если используешь Pull Request для внесения изменений, оформи заголовок и описание PR также на русском языке. В заголовке кратко обозначь суть изменений, а в описании подробно распиши, что сделано и зачем, перечисли ключевые моменты. Придерживайся принятого в проекте шаблона оформления PR, если он существует.

## Развёртывание

- Для подготовки приложения к деплою выполни production-сборку командой `npm run build`. Удостоверься, что сборка завершается успешно и все необходимые файлы появляются в директории `dist`.
- Протестируй собранное приложение локально с помощью `npm run preview`, чтобы убедиться в корректной работе в продакшн-режиме.
- Если для деплоя требуется дополнительная конфигурация (например, установка переменных окружения, настройка прокси или корректного API URL для продакшна), внеси необходимые изменения в файлы настроек (например, файл `.env` или конфигурационные файлы проекта).
- При деплое на сервер или хостинг следуй принятому процессу. Например, скопируй содержимое директории `dist` на целевой сервер (для статического хостинга) или настрой автоматический деплой через CI/CD. При необходимости добавь в репозиторий скрипты или инструкции для деплоя.
- После деплоя проверь работоспособность приложения в боевом окружении и готовность всех функций. При обнаружении проблем сообщи о них и предложи возможные решения.
